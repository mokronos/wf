Tournament Bracket Scheduler

A pure-data event workflow for testing dependency graphs, delayed starts, score signals, and manual overrides.

What It Should Do

Schedule a toy tournament where match N depends on earlier winners. It waits for score submissions, handles disputes, lets a moderator override a result, and emits the final bracket. No real calendar, chat, or game server is required.

Mocked Programs and Services

  • Participant list fixture.
  • Fake clock for match start delays and score deadlines.
  • Score submission signals from test code.
  • Moderator override signal for disputed matches.

Mermaid Diagram

flowchart TD
  A[Seed participants] --> B[Create round one matches]
  B --> C[Wait until match start]
  C --> D[Accept score signal]
  D --> E{Disputed?}
  E -- no --> F[Advance winner]
  E -- yes --> G[Open moderator review]
  G --> H{Override received before deadline?}
  H -- no --> I[Use deterministic tie-break]
  H -- yes --> F
  I --> F
  F --> J{Round complete?}
  J -- no --> C
  J -- yes --> K{Champion known?}
  K -- no --> L[Create next round]
  L --> C
  K -- yes --> M[Publish bracket]

Example Inputs

{
  "tournamentId": "toy-cup",
  "participants": ["Ada", "Grace", "Linus", "Katherine"],
  "matchDurationSeconds": 10,
  "scoreSignals": [
    { "matchId": "r1-m1", "score": { "Ada": 3, "Grace": 1 } },
    { "matchId": "r1-m2", "score": { "Linus": 2, "Katherine": 2 }, "disputed": true },
    { "matchId": "final", "score": { "Ada": 1, "Katherine": 4 } }
  ],
  "moderatorSignals": [
    { "matchId": "r1-m2", "winner": "Katherine", "reason": "fixture override" }
  ]
}
