Release Train Deployment

Engineering workflow for collecting changes, validating release readiness, progressive deployment, and rollback.

What It Should Do

Run a weekly release train across multiple services: freeze candidates, verify tests and migrations, get approvals, deploy progressively, monitor health, roll back when required, and generate release notes from merged work.

Programs and Services Involved

Development
GitHub or GitLab, CI, artifact registry, feature flag service, changelog generator.
Delivery
Kubernetes, Terraform, deployment orchestrator, database migration runner, secrets manager.
Operations
Observability stack, error tracker, status page, incident tool, stakeholder chat channel.

Workflow Diagram

flowchart TD
  A[Release window opens] --> B[Collect merged changes and artifacts]
  B --> C[Create release candidate manifest]
  C --> D[Run tests, scans, migrations dry run, and smoke checks]
  D --> E{Readiness passed?}
  E -- no --> F[Remove failing change or delay train]
  F --> C
  E -- yes --> G[Request release manager approval]
  G --> H[Deploy canary environment]
  H --> I[Run synthetic and business metric checks]
  I --> J{Canary healthy?}
  J -- no --> K[Rollback canary and open blocker]
  K --> F
  J -- yes --> L[Progress to 10, 50, and 100 percent]
  L --> M{Health regression?}
  M -- yes --> N[Pause, rollback affected service, notify channel]
  M -- no --> O[Publish release notes and close train]
  N --> P[Create follow-up incident or hotfix]

Important Branches and Edge Cases

  • One service has a backward-incompatible migration, so deployment order must be constrained.
  • Feature flag rollout should continue after deploy and may be paused independently of code rollout.
  • Canary succeeds technically but business metrics drop, triggering manual review before continuing.
  • Rollback of one service requires compensating database migration or read-only mode.
  • A security patch skips the normal train and runs an expedited approval path.
  • Release notes must exclude internal-only issues while linking public tickets and customer-facing changes.

Example Inputs

  "releaseTrain": "2026.06.21",
  "window": { "start": "2026-06-21T18:00:00Z", "end": "2026-06-21T22:00:00Z" },
  "services": [
    { "name": "billing-api", "artifact": "registry/billing-api:9f21c3", "migration": "2026062118_add_invoice_index.sql" },
    { "name": "customer-web", "artifact": "registry/customer-web:32aa09", "migration": null }
  ],
  "requiredApprovers": ["release-manager@company.example"],
  "healthChecks": ["synthetics", "error_rate", "checkout_success_rate"],
  "rolloutPlan": [10, 50, 100]
}
