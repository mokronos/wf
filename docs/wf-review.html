1The vision

wf is an agent-first workflow automation platform — n8n/Zapier territory, but the LLM agent is the workflow designer, not the runtime. The agent discovers connectors, inspects schemas and auth requirements, composes durable workflows, tests them against fixtures, inspects failures, repairs the definition, and deploys versioned artifacts.

Core principle: structured orchestration, typed integrations, code for computation.

Workflow semantics (triggers, branches, retries, waits, human tasks, compensation, versioning) are first-class engine primitives. Local data manipulation lives in plain TypeScript transform steps — not dozens of specialized UI nodes.

The deliverable is a repeatable workflow artifact, not an expensive LLM execution trace. The LLM designs; the deployed workflow runs deterministically on a small DSL over a durable execution engine (@effect/workflow). Long-term goal: the durability of Temporal-style engines with the accessibility and observability of visual automation tools.

2The v1 spec at a glance

A thin, opinionated SDK over @effect/workflow that hides both syntax boilerplate and semantic footguns.

  • defineStep — colocates execute logic, typed terminal errors, retry policy, and compensation in one definition.
  • Error taxonomy — anything thrown is transient and retried; step.fail(...) / ctx.fail(...) is a typed terminal business failure that triggers a LIFO compensation unwind (saga pattern).
  • Replay safety is the wrapper's biggest job — ctx.run auto-suffixes invocation counters so calling a step twice or in a loop never replays a stale result; ctx.now()/ctx.random() are recorded-and-replayed; a divergence detector fails loudly with NonDeterminismError.
  • Explicit versioning required — executions pin to the version they started on; old versions stay registered until drained.
  • Testing ships in v1 — in-memory runtime, per-step mocking, time-skipping, signal injection, compensation recording. “A saga you can't test is a saga you don't have.”
  • Escape hatch — ctx.effect(...) keeps the full Effect engine reachable; client.history() is the foundation for the eventual no-code UI's execution inspector.
  • No accidental dedup — every start() is fresh unless an explicit idempotencyKey is passed.

Deferred: parallel fan-out (ctx.all), child workflows, continue-as-new, patched()-style migration, the visual/no-code layer.

3Why generators, not async/await

Two halves, both real:

  • The Effect half. The engine is @effect/workflow, and Effect's native sequencing is Effect.gen with yield*. ctx.run honestly returns an Effect, so the escape hatch composes trivially — you never leave Effect-land.
  • The independent half. Async/await durable execution is achievable (Inngest, Restate, Cloudflare Workflows do it), but it creates an unfixable footgun: a bare await fetch(...) outside ctx.run silently breaks replay determinism, and no compiler can forbid it. With generators, yield* only accepts Effects — the type system fences you into the durable primitives.

Effect makes generators the cheap choice; the replay-safety argument makes them the defended choice.

6The seven core gaps

Features that silently require engine semantics or persistence-format decisions — promote these into the spec now.

  1. Cancellation. No client.cancel() in the spec. Every ops story starts with “kill the stuck execution” — and cancel must decide whether to run the compensation stack. status even reports "compensating", but nothing external can trigger it. Most glaring v1 gap.
  2. Triggers. The only entry point is client.start(). Polling triggers need dedup state; trigger identity interacts with idempotency and versioning (a cron firing mid-deploy — which version starts?). Needs at least a trigger binding concept.
  3. Concurrency & rate limiting. “10,000 orders arrived, don't call Stripe 10,000 times concurrently” is a day-one requirement. Must live in the engine's step scheduler — can't be layered on afterward.
  4. History growth + secret redaction. Continue-as-new is deferred (fine for v1), but any polling loop hits history limits. Worse: a bearer token in a step input lives in the history store forever and surfaces in client.history(). Credentials must be references resolved at execution time — a core serialization rule, absent from the spec.
  5. Operator redrive. Durable execution gives transient resume for free, but once terminal failure runs compensation there's no redrive. Decide explicitly: does an exhausted-retries execution park in a resumable state, or fail through to compensation?
  6. Long-lived executions vs deployment velocity. “Keep v1 registered until drained” is fine for minutes-long glue, but the flagship example waits 48 hours and Camunda-class processes run for months. Temporal's patched() and Camunda instance migration exist for this. HITL makes long-lived the common case — this deferred feature gets dragged forward.
  7. Audit hooks. Who started, signaled, cancelled — actor metadata on every external interaction, recorded in history. Trivial now, painful to retrofit, hard requirement for enterprise buyers.

7Human-in-the-loop, done right

The worry: “You basically have to pause/resume in the middle of an activity, which is against the checkpointing model, no?”

The answer: No — you never pause mid-activity. The rule in every durable engine: activities are short and side-effecting; time lives in the workflow. Human waiting is a workflow-level suspension, and ctx.waitForSignal is already exactly the right primitive.

// 1. A normal, fast, checkpointed step — sends the approval request
//    with a link encoding the executionId.
yield* ctx.run(notifyApprover, { executionId, orderId })

// 2. The workflow suspends durably. No process runs, no worker is held —
//    the recorded history IS the checkpoint; the engine keeps one
//    persisted "waiting for signal X or timer Y" row.
const approval = yield* ctx.waitForSignal("managerApproval", ApprovalSchema, {
  timeout: "48 hours",
})

// 3. A webhook / approval UI in the outside world calls:
//    client.signal(executionId, "managerApproval", { approved: true })

This is precisely how everyone does it: Step Functions .waitForTaskToken, Windmill suspend/resume URLs, n8n Wait node, Camunda user tasks (the “task” is a row in a task store; the engine waits on its completion event). Suspension isn't against the checkpointing model — it is the checkpointing model at its purest: a workflow waiting 6 months for a signal costs one database row.

The anti-pattern (correctly sensed): an execute that does await pollUntilHumanApproves() for 48 hours. Holds a worker slot, fights the retry policy, dies on every deploy. Never that.

What's still needed for production-grade HITL

  • Signal buffering. Specify what happens when a signal arrives before the workflow reaches waitForSignal (fast approver, workflow still retrying an earlier step). The engine must buffer per execution — otherwise approvals get lost. Temporal buffers; the spec is silent.
  • A task record, not just a signal. waitForSignal is anonymous. Consider ctx.humanTask({ assignee, form, … }) sugar that creates a task record + waits on its completion signal, so a generic inbox UI is possible. Camunda's per-“task user” pricing shows how much enterprises value this.
  • Signal authorization + audit. Today anyone who knows the execution ID can approve anything. Who may signal, and the recorded actor, are part of the HITL story.
  • Escalation & reminders already work sequentially — no ctx.all needed. Timeout-as-branch gives escalation; reminders are a loop of waitForSignal(timeout: "24 hours") → send reminder → loop (invocation counters make it replay-safe). A point in favor of the v1 scoping.
  • Version-pinning pressure. Every HITL workflow is by definition long-lived — HITL adoption directly drags the deferred patched()-style migration forward.

8Verdict

The core model holds up well: the incumbents' enterprise features are mostly layers to build later, wf's error/compensation/testing semantics beat the iPaaS tier outright, and HITL fits the existing signal primitive cleanly.

The real v1 gaps are operational, not conceptual: cancel, trigger bindings, rate limits, secret references, signal buffering, and actor metadata all touch engine semantics — they belong in the spec now, not in the platform later.
