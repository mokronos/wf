Workflow coverage: wf vs. n8n

How far does today's wf authoring surface — what the agent writes against ctx.*, defineWorkflow, defineError, defineSignal, t.* — go toward covering n8n's node catalog? Each n8n node is scored against the primitives that exist today and against the static parser in wf-to-mermaid.ts (which renders the diagram/UI). External SaaS/service nodes are split into their own section — they all collapse to one pattern you already have a plan for.

What wf exposes today

The headline. The things wf covers cleanly are exactly the things the vision prioritizes: computation (step), durable I/O (activity), waits, signals, retries, compensation, idempotency. The gaps cluster in three places: (1) no trigger surface — workflows are invoked via run(wf, payload), there is no schedule/webhook/event entrypoint yet; (2) the static parser is linear-only — branches, loops and parallelism execute correctly but vanish from the diagram; (3) no item-stream / binary / datastore model — n8n's per-item fan-out and binary blobs don't have a 1:1 home.

Triggers

n8n workflows always start with one. wf has no trigger primitive yet — this is the single biggest gap.
n8n nodeVerdictMaps to / why
Manual TriggerRepresentedThis is just run(wf, payload) / workflow.execute(payload). The typed payload schema is the manual trigger.
Schedule / Cron TriggerNot possibleNo scheduling entrypoint. The engine has DurableClock for in-flow sleep, but nothing that starts a run on a cron. Needs a trigger registry.
Webhook Trigger · Respond to WebhookNot possibleNo HTTP ingress surface. (Resuming a running workflow from outside is solved — that's waitForSignal — but starting one from HTTP is not.)
Error TriggerNot possibleNo "run workflow B when workflow A fails" hook. Per-workflow typed error exists, but not a cross-workflow error subscriber.
Form Trigger · Chat TriggerNot possibleNeed both an HTTP/UI ingress and a rendered form. Mid-flow human input is close (waitForSignal), but the entry form is not.
Email (IMAP) · RSS Feed · Local File · SSE TriggerExternalAll are "poll/subscribe to an external source, then start a run" — a trigger surface plus a connector. Belongs to the integration plan.
Execute Sub-workflow Trigger · Workflow TriggerNot possibleNo sub-workflow invocation primitive exposed in the DSL yet (Effect workflows can call each other under the hood). Vision lists it as first-class — TODO.

Where the two diverge

✅ Where wf is stronger than n8n

  • Durable, deterministic replay. Activities checkpoint exactly-once; the workflow re-runs pure steps for free. n8n persists executions but doesn't replay deterministically.
  • Compensation / saga. compensate runs to undo an activity if the workflow later fails. n8n has no native saga.
  • Idempotency. idempotencyKey makes two runs with the same key one run — first-class.
  • Typed errors. defineError values cross the durable boundary; not stringly-typed.
  • Reviewable artifact. A workflow is versioned TypeScript with annotated steps — diffable, testable, replayable. Not opaque UI JSON.
  • Long waits are free. sleep/waitForSignal hold no resources for days/weeks.

❌ Where n8n is ahead today

  • Triggers. n8n's whole front door — schedule, webhook, form, email, polling. wf has none yet; runs are invoked directly.
  • Visual branch / loop / parallel. wf runs them as JS, but the parser is linear-only, so the diagram is incomplete — the thing reviewers actually look at.
  • 400+ ready connectors. n8n ships auth + schemas for hundreds of apps. wf's registry is planned, not built.
  • Item-based fan-out. n8n auto-runs each node per item; wf requires explicit iteration and has no fan-out execution model.
  • Binary data + datastore. No binary blob model, no built-in Data Table.
  • Zero-code reach. n8n needs no TypeScript; wf is code-first (a deliberate trade for the agent-author model).

🔌 External services & integrations your separate plan

n8n's 400+ app nodes (Slack, Gmail, Google Sheets, Notion, Airtable, Salesforce, HubSpot, Stripe, GitHub, Jira, Telegram, Discord, Twilio, the AWS/Google/Microsoft suites, Postgres/MySQL/MongoDB/Redis, Shopify, and the rest) plus the entire LangChain/AI category (AI Agent, chains, vector stores, embeddings, memory, tools) all collapse to a single shape:

ctx.activity("Slack.postMessage", () => client.chat.postMessage(args), { retry, error, compensate })  — a typed call, behind credentials, from the durable backbone.

So coverage here is not a per-node question — it's one connector-registry question. The vision already calls for "discover available connectors, inspect their schemas and auth requirements." Once that registry + auth/secrets surface exists, every app node becomes a generated, typed activity. The 400+ nodes are breadth (a packaging/curation effort), not new workflow semantics. Databases and vector stores additionally want the missing datastore/state story; LLM/AI nodes additionally sit in tension with the "deterministic deploy" principle and should be isolated activities.
